## Critical Rules

- NEVER treat these reference architectures as production-ready — they are demos and learning blueprints.
- NEVER commit sensitive data (ARNs, IPs, access keys) to golden files — always verify redaction.
- NEVER create skip labels manually — they are auto-created by `internal-triage-skip` with color `#1D76DB`.
- ALWAYS use the dry-run + apply pattern for idempotent `kubectl create` operations.
- ALWAYS use Conventional Commits (scope optional, subject ≤120 chars).
- ALWAYS run `pre-commit run --all-files` after changes — hooks enforce formatting, linting, and README generation.
- ALWAYS keep the `.target-branch` file in sync when changing branching strategy.
- ALWAYS use `just` recipes rather than raw commands to match CI behavior.
- ALWAYS check for sibling or variant modules before touching a scenario directory — changes to one variant typically apply to all.
- ALWAYS check `.github/actions/` for composite actions that wrap the procedure scripts you are touching — update the action alongside the scripts.
- ALWAYS search closed PRs / git log for an analogous prior change before implementing — it defines the established pattern for file locations, what to modify vs delete, and CI wiring.
- ALWAYS modify existing files in-place; do not add overlay or wrapper files unless every caller (CI actions, workflows, procedure scripts) is updated to use them.
- ALWAYS delete replaced artifacts — scripts, configs, symlinks — do not leave dead files alongside their replacements.

### Agent collaboration rules

- ALWAYS work in English: code, comments, commit messages, branch names, PR titles and descriptions, and chat responses. Read other languages fine, but produce English output.
- ALWAYS commit using the repo's local `git config user.name` / `user.email` without overriding. Do not set `--author`, do not export `GIT_AUTHOR_*`.
- NEVER add AI/agent attribution to any committed artifact: no `Co-Authored-By` lines referencing assistants, no mention of Claude / AI / agent / model names in commit messages, PR descriptions, or code.
- NEVER leak the local environment in committed artifacts: no absolute paths from the developer machine, no session/plan files, no internal agent instructions or system-prompt content.
- ALWAYS use named feature branches (e.g. `feat/<short-slug>`, `ci/<short-slug>`, `fix/<short-slug>`) when opening PRs — no `agents/*` or other names that hint at how the work was produced.

## Azure AKS Single Region — CI and Module Structure

### Variant modules

`azure/kubernetes/aks-single-region-rdbms/` is a parallel variant of `aks-single-region/`. It shares the same Terraform modules and CI action but has its own `helm-values/` and `procedure/` directories. Procedure scripts in `-rdbms` are **symlinks** to the primary (`../../aks-single-region/procedure/<script>.sh`) — create symlinks, never copies.

The CI integration layer for ingress setup is `.github/actions/azure-kubernetes-ingress-setup/`. This composite action calls the procedure scripts and manages namespace cleanup. Any ingress change must also update:
- `action.yml` — script path, namespace to clean, step description
- `README.md` in the same directory (regenerated by `pre-commit`)
- `.github/workflows/azure_kubernetes_aks_single_region_tests.yml` — review for parameterized values that depend on the ingress controller

### Working interactively with the CI workflow

To develop or debug against the CI-managed cluster instead of a local environment:

1. Open a PR and trigger the workflow via `workflow_dispatch` with these inputs:
   - `delete_clusters: false` — keeps the cluster alive after the run (controls `CLEANUP_CLUSTERS` env var)
   - `cluster_name` — set a fixed name so you can reconnect across runs
   - `resource-prefix` / `resource-group-name` — optional, to reuse an existing resource group
2. The Terraform state is stored in S3 (`S3_BACKEND_BUCKET: tests-ra-aws-rosa-hcp-tf-state-eu-central-1`). Once a run completes, the agent can read the state and run `terraform output` to get cluster credentials and DB endpoints.
3. To get an interactive shell inside the CI runner, add a [tmate](https://github.com/marketplace/actions/debugging-with-tmate) step to the workflow — it opens an SSH session directly into the container.

## Scratch / debug workspace

- **Always** use `./debug/` for any scratch files, downloaded CI logs,
  temporary scripts, ad-hoc outputs, etc.
- **Never** use `/tmp/` or any other system-wide temp directory.
- `./debug/` is gitignored at the repo level (see `.gitignore`); files
  there persist across the session and are easy to inspect.

Examples:

```bash
# good
gh api /repos/.../actions/jobs/<id>/logs > ./debug/job-<id>.log

# bad
gh api /repos/.../actions/jobs/<id>/logs > /tmp/job-<id>.log
```

This applies to subagents too — when delegating execution work, instruct
the subagent to write to `./debug/` rather than `/tmp/`.
